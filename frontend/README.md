# VisaJobFinder frontend (engine port)

Vite + React + TypeScript + Vitest. This package currently contains a
**client-side TypeScript port of the deterministic visa engine** so the browser
paste-checker can classify pasted job ads with no server round-trip. There is no
real UI yet (a later worker builds it); `src/App.tsx` is a placeholder shell.

The app is **static and local-only**: no analytics, no telemetry, no network
libraries. The only data it loads is a static sponsor register
(`public/sponsors.json`).

## Engine layout (`src/engine/`)

| File | Ports from (Python source of truth) |
| --- | --- |
| `types.ts` | `pipeline/classifier/models.py` (JobInput, UserContext, EmployerMatch, ClassificationResult, PhraseSignal, ALLOWED_LABELS, ALLOWED_VISA_SITUATIONS) |
| `phraseScanner.ts` | `pipeline/classifier/phrase_scanner.py` (PHRASE_RULES verbatim + overlap resolution) |
| `normalise.ts` | `pipeline/sponsor_register/normalise.py` |
| `sequenceMatcher.ts` | `difflib.SequenceMatcher.ratio()` (Ratcliff/Obershelp) used by the matcher |
| `sponsorMatcher.ts` | `pipeline/sponsor_register/matcher.py` (scoring + confidence bands) |
| `engine.ts` | `pipeline/classifier/engine.py` (classify_job + analyse_job) |
| `sponsorData.ts` | loads the static `sponsors.json` into a `SponsorMatcher` |

> **These files are a PORT and MUST stay in sync with the Python source of
> truth.** The Python engine is authoritative. If you change a Python rule,
> regenerate the golden fixture (below) and re-run the parity test. If the TS
> port no longer matches, fix the TS port — never hand-edit the golden file.

## SequenceMatcher equivalence

The Python matcher uses `difflib.SequenceMatcher(None, a, b).ratio()` in three
places. `sequenceMatcher.ts` is a faithful re-implementation of CPython's
algorithm: `find_longest_match` (via the b2j index), recursive
`get_matching_blocks`, and `ratio() = 2*M / (len(a) + len(b))`. `isjunk` is
`None` (as in the Python engine) so there is no junk set, and the autojunk
heuristic only fires for sequences ≥ 200 chars (never true for employer names),
but it is implemented anyway for exactness. Confidence values match Python to at
least 3 decimal places across the whole parity corpus.

## Regenerating the golden parity data

The golden fixture and the static sponsor register are both generated by a
Python script that runs the **live** Python engine, so they always reflect the
current Python rules.

```bash
# from the repo root (needs the pipeline deps installed)
python scripts/gen_parity_golden.py
```

This writes:

- `frontend/src/engine/__fixtures__/parity_golden.json` — every eval case
  (real + sample, expanded per visa situation exactly as
  `pipeline/eval/run_eval.py` does) with the Python label, the set of evidence
  categories, and the sponsor match (matched_name, confidence_band, is_match,
  confidence).
- `frontend/public/sponsors.json` and
  `frontend/src/engine/__fixtures__/sponsors.json` — the sponsor register from
  `data/sponsor_register/sample_sponsors.csv`.

## Running the parity test

```bash
cd frontend
npm install
npm run test     # vitest run — includes src/engine/parity.test.ts
npm run build    # tsc -b && vite build
```

`parity.test.ts` loads `parity_golden.json`, runs the TS engine over the same
inputs, and asserts, for every case:

- the TS **label** equals the Python label;
- the **set of evidence categories** equals Python's;
- the sponsor **matched_name / confidence_band / is_match** equal Python's;
- the sponsor **confidence** is within 3 dp of Python's.

## Conservative language

The engine reuses the exact reason / limitation / verification strings from the
Python engine. It never emits copy like "eligible", "safe", "can apply", or
"will sponsor". See `CLAUDE.md` for the product language rules.
